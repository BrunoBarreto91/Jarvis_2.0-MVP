🛡️ PROTOCOLO DE RESILIÊNCIA: JARVIS U.IEste documento define os padrões obrigatórios para garantir que a interface do Jarvis 2.0 seja "à prova de falhas" em relação aos dados provenientes do n8n.1. Tratamento de Dados (Camada de Serviço)Sanitização Mandatória: Toda chamada à API no api.ts deve passar por um wrapper de segurança.Null Safety: Nunca assuma que um campo existe. Use encadeamento opcional (task?.title) e valores padrão (task?.title ?? 'Sem descrição').Array Guard: Sempre verifique Array.isArray(response). Se a API retornar um objeto único, o serviço deve convertê-lo em [objeto] antes de entregar ao componente.2. Experiência do Usuário (TDAH-Friendly)Loading States: Todo componente que consome dados deve exibir um Skeleton ou Spinner do Shadcn/UI enquanto aguarda o n8n. Interfaces vazias ou "travadas" aumentam a ansiedade.Empty States: Se não houver tarefas, exiba uma mensagem clara e motivadora, nunca uma tela branca.Error Boundaries: Utilize o componente ErrorBoundary.tsx para isolar falhas. Se o Kanban crashar, a Sidebar e o restante da página devem permanecer funcionais.3. Mapeamento de Linguagem (PT-BR -> EN)O banco de dados (MySQL) utiliza colunas em Português para facilitar a leitura do n8n.A UI deve operar exclusivamente com nomes de propriedades em Inglês (CamelCase).O Mapper no api.ts é o único local autorizado para realizar essa tradução.
